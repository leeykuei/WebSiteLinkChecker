**建立日期**：2026-03-07  
**狀態**：草稿  
**輸入**：用戶描述："請增強目前的網站連結檢查規格，加入「即時檢查進度可視化」需求。新增需求：1. 系統在執行網站連結檢查時，應即時顯示檢查進度。2. 系統應顯示目前正在掃描的頁面 URL。3. 系統應顯示正在檢查的超連結 URL。4. 系統應顯示掃描統計資訊，包括：已發現頁面數量、已處理頁面數量、已檢查連結數量、已發現失效連結數量。5. 系統應提供進度指示（例如百分比或 progress bar）。6. 系統應在 Console 持續輸出進度資訊，避免使用者等待時沒有任何回饋。"

## 使用者情景與測試 *(必填)*

### 使用者故事 1 - 基本靜態連結驗證 (優先級：P1)

用戶需要一個簡單的工具，輸入一個網站 URL，程式能自動提取頁面上所有 HTML `<a>` 標籤中的連結，並檢查每個連結的 HTTP 狀態碼是否為 200（有效）。對於無法存取的連結，能捕捉錯誤代碼（如 404、500、timeout）。

**為何此優先級**：這是核心功能，滿足基本的連結檢查需求。即使只實現這個故事，也能為用戶提供實用的工具。

**獨立測試**：可以獨立測試靜態 HTML 頁面的連結提取與驗證，不需依賴 JavaScript 檢測功能或進度可視化。

**接受準則**：

1. **Given** 用戶提供一個有效的 URL 如 `https://www.example.com`，**When** 程式執行，**Then** 程式應提取頁面上所有 `<a href="...">` 連結，並回傳每個連結的 HTTP 狀態碼
2. **Given** 頁面包含多個連結，**When** 檢查連結，**Then** 應能正確識別有效連結（狀態碼 200）與無效連結（4xx、5xx）
3. **Given** 連結檢查失敗（如 timeout、DNS 解析失敗），**When** 程式執行，**Then** 應捕捉錯誤訊息並記錄具體的失敗原因

---

### 使用者故事 2 - 即時進度可視化 (優先級：P1)

用戶在執行連結檢查時，需要即時看到掃描進度，包括目前正在處理的頁面 URL、正在檢查的連結 URL，以及統計資訊（已發現頁面數、已處理頁面數、已檢查連結數、失效連結數）。系統應持續更新顯示，提供清晰的進度指示器，避免用戶在等待時感到不確定。

**為何此優先級**：對於包含大量連結的網站，檢查過程可能耗時數分鐘甚至更長。即時進度回饋能顯著提升用戶體驗，讓用戶了解工具正在運作且尚未停滯，同時能預估完成時間。

**獨立測試**：可以獨立測試進度顯示邏輯，驗證資訊更新頻率、統計數據準確性、顯示格式的可讀性，無需完整的連結檢查功能。

**接受準則**：

1. **Given** 程式正在掃描包含多個頁面的網站，**When** 掃描進行中，**Then** 系統應持續顯示目前正在掃描的頁面 URL
2. **Given** 程式正在檢查個別連結，**When** 檢查進行中，**Then** 系統應顯示目前正在檢查的連結 URL
3. **Given** 掃描包含 100 個連結的網站，**When** 已檢查 50 個連結且總連結數未變，**Then** 進度指示器應顯示 50% 的完成度；若檢查過程中發現額外 50 個新連結（總數變為 150），**Then** 進度百分比應動態調整為約 33%（50/150）
4. **Given** 掃描過程中，**When** 發現新頁面或新連結，**Then** 統計資訊應即時更新（已發現頁面數、已檢查連結數）
5. **Given** 掃描過程中發現失效連結，**When** 檢查完成該連結，**Then** 系統應立即追加新行顯示失效連結的 URL 和狀態碼，且失效連結計數應即時增加
6. **Given** 用戶正在觀察進度顯示，**When** 超過 2 秒沒有更新，**Then** 用戶應能看到系統仍在運作的指示（如動畫或最後更新時間戳）

---

### 使用者故事 3 - 動態菜單連結檢測 (優先級：P2)

部分網站（如現代前端框架 React/Vue 搭建）的菜單或導覽連結是透過 JavaScript 動態渲染的。用戶需要工具能夠啟動真實瀏覽器引擎（支援 JavaScript 執行），等待頁面完全加載後再提取所有連結。

**為何此優先級**：許多現代網站採用動態渲染，靜態提取無法獲得完整連結列表。此功能擴展了工具的適用範圍。

**獨立測試**：可以在啟動 JavaScript 引擎的分支上進行測試，提供動態渲染頁面的檢查能力；即使靜態連結驗證分支部署，此分支也可獨立使用。

**接受準則**：

1. **Given** 頁面包含 JavaScript 動態生成的菜單連結，**When** 程式使用瀏覽器引擎執行，**Then** 程式應等待 DOM 完全加載，提取所有動態產生的 `<a>` 標籤
2. **Given** 用戶選擇「動態檢測模式」，**When** 程式執行，**Then** 應使用瀏覽器引擎啟動，執行 JavaScript 並提取完整連結清單
3. **Given** 動態連結檢測完成，**When** 與靜態提取結果比對，**Then** 應能識別只在動態渲染時出現的連結

---

### 使用者故事 4 - CSV 報表生成 (優先級：P2)

用戶需要將連結檢查結果輸出為結構化的 CSV 格式報表，包含：連結 URL、HTTP 狀態碼、檢查耗時、錯誤訊息（如適用）。報表應易於在 Excel 等工具中開啟和分析。

**為何此優先級**：報表是使用者與檢查結果互動的主要方式，便於後續人工審查和文檔化。此功能與 P1 併行實現價值最高。

**獨立測試**：可以獨立測試 CSV 生成邏輯，驗證輸出格式、字符編碼、欄位完整性。

**接受準則**：

1. **Given** 檢查完成 N 個連結，**When** 生成報表，**Then** CSV 應包含 N 列資料，每列包含：URL、狀態碼、耗時、錯誤訊息
2. **Given** 報表包含失效連結（HTTP != 200），**When** CSV 生成，**Then** 失效連結應清楚標註，便於篩選和排序
3. **Given** 報表內容包含中文字符，**When** 儲存為 CSV，**Then** 應使用 UTF-8 編碼確保字符正確顯示

---

### 邊界情況

- 當 URL 無效或格式錯誤時（如 `htp://example` 或空字符串），程式應提示錯誤而不是崩潰
- 當頁面重定向（HTTP 301/302）時，程式應跟隨重定向並檢查最終連結
- 當連結超時（>10 秒）時，應記錄為超時錯誤而不是持續等待
- 當頁面包含相對連結（如 `/page`、`../resource`）時，應正確解析相對於基礎 URL 的絕對路徑
- 當頁面的某個連結指向自身（如 `https://example.com/page#section`）時，應能正確驗證
- 當網站包含超過 10,000 個連結時，進度顯示應保持流暢且不影響檢查效能
- 當檢查過程中網路中斷時，進度顯示應清楚標示錯誤狀態並保留已完成的部分進度
- 當用戶中途取消檢查時，應顯示已完成的進度統計並正確儲存部分結果
- 當發現新連結導致總數增加時，進度百分比應正確反映新的比例（可能暫時下降）
- 當終端機不支援 ANSI 跳脫序列時，應降級為追加新行模式避免顯示錯誤
- 當失效連結數量過多（如超過 100 個）時，應考慮限制即時顯示數量或提供摘要模式

## 需求 *(必填)*

### 功能需求

#### 核心連結檢查功能

- **FR-001**：系統 MUST 接受命令行輸入 URL，格式為 `python link_checker.py --url <URL>`
- **FR-002**：系統 MUST 解析 HTML 頁面並提取所有 `<a href="...">` 標籤中的 URL
- **FR-003**：系統 MUST 對每個提取的 URL 進行 HTTP HEAD 或 GET 請求，取得 HTTP 狀態碼
- **FR-004**：系統 MUST 支援重試機制（最多 3 次重試，指數退避），處理暫時的網路失敗
- **FR-005**：系統 MUST 支援可配置的 HTTP 請求 timeout（預設 10 秒，範圍 3-60 秒）
- **FR-006**：系統 MUST 使用非同步方式並發檢查連結，支援可配置的並發數（預設 10，範圍 1-50）
- **FR-007**：系統 MUST 可選地使用瀏覽器引擎執行 JavaScript，等待 DOM 完全載入後提取連結
- **FR-008**：系統 MUST 將檢查結果輸出為 CSV 檔案，包含欄位：URL、狀態碼、檢查耗時（毫秒）、錯誤訊息
- **FR-009**：系統 MUST 記錄詳細的日誌到檔案和控制台，日誌層級包括 DEBUG、INFO、WARNING、ERROR、CRITICAL
- **FR-010**：系統 MUST 支援相對 URL 的解析，轉換為絕對 URL 後檢查
- **FR-011**：系統 MUST 跟隨 HTTP 重定向（301、302、307、308）並檢查最終目標
- **FR-012**：系統 MUST 捕捉並記錄所有異常（網路錯誤、超時、DNS 失敗等）的詳細資訊

#### 即時進度可視化功能

- **FR-013**：系統 MUST 在檢查過程中持續顯示目前正在掃描的頁面 URL
- **FR-014**：系統 MUST 在檢查過程中顯示目前正在檢查的個別連結 URL
- **FR-015**：系統 MUST 即時顯示掃描統計資訊，包括：
  - 已發現頁面總數
  - 已處理頁面數量
  - 已檢查連結總數
  - 已發現失效連結數量
- **FR-016**：系統 MUST 提供整體進度指示器，以圖形進度條搭配百分比數字的形式顯示完成度
- **FR-017**：系統 MUST 每秒至少更新一次進度顯示，確保用戶能感知系統正在運作
- **FR-018**：系統 MUST 在進度顯示中包含時間資訊，如已執行時間和預估剩餘時間（基於已處理連結的平均耗時計算）
- **FR-019**：系統 MUST 在檢查完成時顯示總結資訊，包括總耗時、總連結數、有效連結數、失效連結數
- **FR-020**：系統 MUST 使用單行覆寫方式更新進度資訊，避免終端機被大量訊息淹沒
- **FR-021**：系統 MUST 在發現失效連結時立即追加新行顯示其 URL 和狀態碼，確保資訊不被進度更新覆蓋

### 關鍵實體

- **連結 (Link)**：代表頁面上的單個超連接，包含原始 URL、解析後的絕對 URL、HTTP 狀態碼、檢查耗時、錯誤訊息
- **檢查結果 (CheckResult)**：包含頁面 URL、檢查開始/結束時間、連結總數、有效連結數、無效連結數、超時連結數
- **配置 (Configuration)**：包含 timeout 時間、並發限制、重試次數、是否啟用 JavaScript 檢測等參數
- **進度狀態 (ProgressState)**：包含目前處理的頁面/連結 URL、已發現/已處理的頁面數、已檢查/失效的連結數、完成百分比、已執行時間

## 假設 (Assumptions)

- 用戶提供的 URL 為有效的 HTTP/HTTPS 地址
- 目標網站允許程式進行連結檢查請求（無嚴格的 rate limiting 或反爬蟲阻擋）
- 網路連線穩定，至少支援 HTTP/1.1
- Python 環境版本 ≥ 3.9，且已安裝必要的相依套件（aiohttp、Playwright 等）
- 用戶使用支援 ANSI 跳脫序列的終端機（用於進度顯示更新）
- 進度顯示採用行緩衝輸出，確保在標準輸出環境中能正確顯示

## 已決定事項

- **決定 1（瀏覽器引擎選擇）**：使用 Playwright 
  - 理由：Playwright 在速度、跨瀏覽器支援與安裝便利性上兼具優勢，能可靠處理現代前端框架的動態渲染需求；預設在啟用動態檢測模式時使用 Playwright 啟動 headless 瀏覽器擷取完整 DOM。

- **決定 2（有效連結的 HTTP 狀態碼定義）**：接受所有 2xx 狀態碼視為有效
  - 理由：寬鬆模式可涵蓋多種成功回應（例如 204、206），減少誤判，同時保留將來提供自訂狀態碼清單的彈性。

- **決定 3（CSV 報表內容範圍）**：預設輸出完整報表（所有連結）
  - 理由：完整報表便於稽核與驗證；實作上亦可提供 `--report-type failures` 選項以只輸出失效連結。

- **決定 4（進度更新頻率）**：每秒至少更新一次
  - 理由：平衡用戶體驗與系統效能。每秒更新能提供流暢的即時回饋，同時不會造成過多的 I/O 開銷。

## Clarifications

### Session 2026-03-07

- Q: 進度資訊在終端機中的顯示方式應採用何種更新策略？ → A: 選項 B - 單行覆寫更新（使用 `\r` 或 ANSI 跳脫序列）
- Q: 進度條的視覺呈現形式應採用哪種方式？ → A: 選項 C - 圖形進度條 + 百分比數字（如「[=====>    ] 45%」）
- Q: 進度百分比的計算應基於什麼基準？ → A: 選項 B - 基於已發現連結總數動態計算
- Q: 發現失效連結時應如何即時通知用戶？ → A: 選項 B - 追加新行顯示失效連結詳情（URL + 狀態碼）
- Q: 預估剩餘時間應採用何種計算方式？ → A: 選項 B - 基於已處理連結的平均耗時

- **決定 5（進度顯示更新策略）**：使用單行覆寫更新方式
  - 理由：單行覆寫（使用 `\r` 回車符或 ANSI 跳脫序列）是命令列工具的業界標準做法，提供流暢的視覺效果且不會讓終端機被大量訊息淹沒。此方式在 wget、npm、pip 等常用工具中廣泛採用，用戶熟悉度高。重要事件（如發現失效連結、錯誤）仍可追加新行保留在歷史記錄中，而一般進度資訊則持續覆寫同一行。

- **決定 6（進度條視覺呈現）**：使用圖形進度條搭配百分比數字
  - 理由：結合圖形進度條（如「[=====>    ]」）和百分比數字能同時提供直觀的視覺反饋和精確的量化資訊。這是現代命令列工具（如 pip、cargo、npm）的標準做法，用戶能一眼看出大致進度（圖形）並獲得準確數值（百分比），兼具易讀性與精確度。

- **決定 7（進度百分比計算基準）**：基於已發現連結總數動態計算
  - 理由：在網站爬取場景中，新頁面和新連結會持續被發現，總數是動態變化的。基於「已檢查連結數 / 已發現連結總數」計算進度更符合實際情況，百分比會隨著發現新連結而動態調整（可能暫時下降），但能準確反映「已處理/已知工作」的比例。這種動態性是網頁爬蟲工具的常見行為，用戶能理解，且實作邏輯清晰明確。

- **決定 8（失效連結即時通知）**：追加新行顯示失效連結詳情
  - 理由：失效連結是關鍵資訊，應永久保留在終端機輸出中供用戶查看。當發現失效連結時，追加新行顯示其 URL 和狀態碼（不被進度更新覆蓋），確保用戶能看到每個失效連結的詳細資訊。一般進度資訊繼續在固定行覆寫更新。這種混合策略（重要事件追加新行 + 一般進度單行覆寫）在 git、rsync 等工具中廣泛使用，能平衡即時回饋和重要事件通知需求。

- **決定 9（預估剩餘時間計算方式）**：基於已處理連結的平均耗時
  - 理由：使用「剩餘連結數 × 平均每連結耗時」進行線性預估，計算邏輯簡單清晰。雖然在動態發現新連結時預估會波動，但用戶能理解其侷限性，且符合 SC-012（預估誤差 ≤ 30%）的要求。相較於複雜演算法（如移動平均或機器學習），此方式實作簡單、效能開銷低，適合命令列工具的即時計算需求。
