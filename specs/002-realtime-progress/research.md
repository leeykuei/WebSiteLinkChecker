# Research: 即時檢查進度可視化技術決策

**Feature**: 002-realtime-progress  
**Date**: 2026-03-07  
**Status**: Phase 0 Complete

## 研究概述

本研究聚焦於命令列進度顯示的最佳實踐、Python 終端機控制技術、非同步架構下的進度追蹤模式，以及確保進度顯示不影響核心檢查效能的實作策略。

## 1. 終端機進度顯示技術

### 決定：使用 ANSI 跳脫序列進行單行覆寫

**研究發現**：
- **ANSI 跳脫序列**是跨平台終端機控制的業界標準
- `\r`（回車符）可將游標移至行首，搭配覆寫實現單行更新
- ANSI 跳脫碼 `\033[K` 可清除游標右側至行尾的內容，避免殘留字符
- Python 標準庫無需額外相依套件即可使用

**選擇理由**：
- 符合 pip、npm、cargo 等主流 CLI 工具的使用習慣
- 實作簡單，效能開銷低（僅為字符串輸出）
- 支援 Windows 10+、Linux、macOS 現代終端機

**替代方案**：
- ❌ `curses` 函式庫：過於重量級，需要完整終端機控制權，不適合輕量進度顯示
- ❌ 純 `\r` 回車符：無法清除行尾殘留字符，顯示效果差
- ❌ 第三方套件（如 `tqdm`、`rich`）：引入額外相依套件，增加專案複雜度

**降級策略**：
當檢測到終端機不支援 ANSI 跳脫序列時（檢查 `sys.stdout.isatty()` 和環境變數），降級為追加新行模式。

**參考資料**：
- ANSI escape codes: https://en.wikipedia.org/wiki/ANSI_escape_code
- Python `sys.stdout` 文檔

## 2. 進度條視覺呈現設計

### 決定：圖形進度條 + 百分比數字 + 統計資訊

**研究發現**：
- 圖形進度條提供直觀視覺反饋，百分比數字提供精確量化
- 統計數字（已檢查/總數）提供具體進度資訊
- 預估剩餘時間（ETA）幫助用戶規劃等待時間

**顯示格式範例**：
```
正在檢查: https://example.com/page1
[=========>          ] 45% | 已檢查: 450/1000 | 失效: 12 | 用時: 01:23 | 剩餘: ~01:50
```

**設計元素**：
- **進度條**：固定寬度（20 字符），使用 `=` 表示已完成，`>` 表示當前位置，空格表示未完成
- **百分比**：顯示到整數位（如 45%）
- **統計資訊**：已檢查/總數、失效連結數、已執行時間、預估剩餘時間
- **目前 URL**：單獨一行顯示，可能需要截斷過長 URL（≤80 字符）

**選擇理由**：
- 符合使用者故事 2 的所有接受準則
- 資訊密度適中，易於快速掃視
- 與 wget、pip 等工具的顯示風格一致

## 3. 進度百分比計算策略

### 決定：基於已發現連結總數動態計算

**研究發現**：
- 網站爬取過程中，新頁面和連結持續被發現，總數動態變化
- 公式：`進度百分比 = (已檢查連結數 / 已發現連結總數) × 100%`
- 當發現新連結時，總數增加，百分比可能暫時下降

**設計考量**：
- **優點**：準確反映「已知工作」的完成度，計算邏輯清晰
- **缺點**：百分比可能「倒退」（如從 50% 降至45%），但符合實際情況
- **用戶溝通**：在文檔中說明此行為是正常的，反映了持續發現新連結的動態過程

**選擇理由**：
- 符合規格中的決定 7（進度百分比計算基準）
- 避免誤導用戶（如果只基於初始連結數，會忽略後續發現的連結）
- 實作簡單，無需預測演算法

**邊界情況處理**：
- 當總數為 0 時，顯示 0%（避免除以零錯誤）
- 當總數不斷增加時，清楚標示「發現中...」狀態

## 4. 預估剩餘時間計算

### 決定：基於已處理連結的平均耗時進行線性預估

**研究發現**：
- 公式：`預估剩餘時間 = (未檢查連結數) × (平均每連結耗時)`
- 平均耗時 = `總檢查耗時 / 已檢查連結數`
- 簡單線性預估在多數情況下誤差 ≤ 30%

**設計考量**：
- **優點**：實作簡單，計算開銷低，無需複雜統計演算法
- **缺點**：無法適應連結檢查速度的變化（如網路速度波動）
- **顯示格式**：`剩餘: ~MM:SS`，使用 `~` 前綴標示為估計值

**選擇理由**：
- 符合規格中的決定 9（預估剩餘時間計算方式）
- 符合 SC-012 成功準則（預估誤差 ≤ 30%）
- 相較於移動平均或機器學習模型，線性預估的簡潔性和效能更適合 CLI 工具

**改進空間**（未來可選優化）：
- 使用指數移動平均（EMA）平滑速度波動
- 區分不同域名的平均耗時（某些網站可能較慢）

## 5. 非同步架構下的進度追蹤模式

### 決定：事件驅動的進度狀態更新

**研究發現**：
- 在 asyncio 架構中，進度更新必須避免阻塞 event loop
- 使用共享狀態物件（`ProgressState`）追蹤進度，由事件觸發更新
- 進度顯示更新採用定時任務（每秒觸發一次），與連結檢查並行

**設計模式**：
```python
class ProgressState:
    """進度狀態追蹤器"""
    def __init__(self):
        self.discovered_links = 0
        self.checked_links = 0
        self.failed_links = 0
        self.current_url = ""
        self.start_time = time.time()
        self.lock = asyncio.Lock()  # 確保線程安全
    
    async def update(self, **kwargs):
        async with self.lock:
            # 更新狀態
            ...
```

**更新機制**：
1. **連結檢查完成時**：觸發 `checked_links += 1`，更新 `failed_links`（如失效）
2. **發現新連結時**：觸發 `discovered_links += 1`
3. **定時顯示任務**：每秒讀取狀態並渲染進度行
4. **失效連結通知**：立即追加新行顯示（不等待定時任務）

**選擇理由**：
- 符合憲法的非同步處理原則
- 避免進度更新阻塞 event loop，確保效能（開銷 ≤ 5%）
- 清晰的責任分離：檢查邏輯負責更新狀態，顯示邏輯負責渲染

**替代方案**：
- ❌ 每次檢查完成後立即更新顯示：頻繁 I/O 操作可能影響效能
- ❌ 使用多線程：增加複雜度，asyncio 架構不需要多線程

## 6. 失效連結即時通知機制

### 決定：追加新行顯示失效連結詳情（URL + 狀態碼）

**研究發現**：
- 失效連結是關鍵資訊，必須永久保留在終端機輸出中
- 混合顯示策略：一般進度單行覆寫 + 重要事件追加新行

**顯示範例**：
```
❌ 失效連結: https://example.com/broken-page (HTTP 404)
[=========>          ] 45% | 已檢查: 450/1000 | 失效: 12 | ...
```

**選擇理由**：
- 符合規格中的決定 8（失效連結即時通知）
- 確保用戶不會miss重要的失效連結資訊
- 與 git、rsync 等工具的混合顯示策略一致

**格式設計**：
- 使用 `❌` emoji 或 `[FAIL]` 文字標記（根據終端機支援情況）
- 顯示完整 URL（或截斷至 80 字符）
- 顯示 HTTP 狀態碼和錯誤訊息（如 timeout、DNS 失敗）

## 7. 效能與資源開銷控制

### 決定：進度顯示開銷限制在 5% 以內

**研究發現**：
- 進度顯示的主要開銷來自：字符串格式化、終端機 I/O 操作
- 定時更新（每秒一次）遠低於連結檢查頻率（每秒 10-50 次），開銷可控
- 避免在熱路徑中進行複雜計算（如實時排序、統計）

**優化策略**：
1. **減少 I/O 頻率**：僅每秒更新一次進度行，批量渲染
2. **預格式化字符串**：使用 f-string 而非多次字符串拼接
3. **懶惰計算**：僅在顯示時計算百分比和預估時間，不在更新時計算
4. **異步寫入**：使用 `sys.stdout.write()` 而非 `print()`，避免額外刷新

**測試驗證**：
- 使用 `pytest-benchmark` 測量進度更新函式的耗時
- 在實際檢查場景中對比啟用/禁用進度顯示的總耗時差異
- 目標：開銷 ≤ 5%（符合 SC-013 成功準則）

## 8. 終端機相容性與降級策略

### 決定：支援 ANSI + 降級至追加新行模式

**研究發現**：
- 現代終端機（Windows 10+、Linux、macOS）廣泛支援 ANSI 跳脫序列
- 某些環境（CI/CD 管道、重定向輸出）不支援 ANSI
- 可通過 `sys.stdout.isatty()` 和環境變數檢測終端機能力

**檢測邏輯**：
```python
def supports_ansi() -> bool:
    """檢查終端機是否支援 ANSI 跳脫序列"""
    if not sys.stdout.isatty():
        return False
    if os.getenv('TERM') == 'dumb':
        return False
    if sys.platform == 'win32':
        # Windows 10+ 預設支援，但檢查版本
        return sys.getwindowsversion().build >= 10586
    return True
```

**降級行為**：
- **支援 ANSI**：使用單行覆寫模式顯示進度
- **不支援 ANSI**：每次更新追加新行（如每 5 秒一次，減少輸出量）
- **重定向輸出**：禁用進度顯示，僅輸出失效連結和最終摘要

**選擇理由**：
- 確保工具在各種環境中都能正常運作
- 符合邊界情況處理要求（規格中已識別）

## 9. 用戶中途取消處理

### 決定：捕捉 SIGINT/KeyboardInterrupt 並顯示部分結果

**研究發現**：
- 使用者可能在檢查過程中按 Ctrl+C 中途取消
- 必須捕捉 `KeyboardInterrupt` 異常並優雅關閉
- 顯示已完成的統計資訊，保存部分結果至 CSV

**實作邏輯**：
```python
try:
    await run_link_checks()
except KeyboardInterrupt:
    logger.warning("檢查被用戶中斷")
    # 顯示部分統計
    print(f"\n已檢查: {state.checked_links}/{state.discovered_links}")
    print(f"失效連結: {state.failed_links}")
    # 保存部分結果
    write_csv(partial_results)
    sys.exit(130)  # SIGINT 的標準退出碼
```

**選擇理由**：
- 符合憲法的穩定性原則（優雅關閉）
- 符合邊界情況處理要求（用戶中途取消）
- 提供良好的用戶體驗，避免資料遺失

## 10. 大量失效連結的顯示策略

### 決定：限制即時顯示數量，提供摘要模式

**研究發現**：
- 當失效連結數量過多（如 >100 個）時，追加新行會淹沒終端機
- 需要平衡即時通知和輸出可讀性

**顯示策略**：
- **預設模式**：失效連結 ≤ 50 個時，全部即時顯示
- **摘要模式**：失效連結 > 50 個時，僅顯示前 50 個 + 提示「還有 N 個失效連結，詳見 CSV 報表」
- **可配置**：透過命令列參數 `--max-failures-display` 調整顯示上限

**選擇理由**：
- 符合邊界情況處理要求（大量失效連結）
- 平衡即時回饋和輸出可讀性
- 確保終端機不被過多訊息淹沒

## 總結與建議

### 核心技術決策

| 決策領域 | 選擇方案 | 替代方案 | 理由 |
|---------|---------|---------|------|
| 終端機控制 | ANSI 跳脫序列 | curses / 第三方套件 | 輕量、跨平台、零相依 |
| 進度條形式 | 圖形條 + 百分比 | 僅百分比 / spinner | 直觀且精確 |
| 百分比計算 | 動態總數 | 初始總數 / 不顯示 | 準確反映實際進度 |
| 預估時間 | 線性平均 | 移動平均 / ML | 簡單且誤差可接受 |
| 架構模式 | 事件驅動 | 多線程 / 同步 | 符合 asyncio 架構 |
| 失效通知 | 追加新行 | 僅計數 / 僅CSV | 永久可見且重要 |
| 效能控制 | 定時更新（1秒） | 即時更新 | 開銷 ≤ 5% |
| 相容性 | 支援 + 降級 | 僅 ANSI | 確保各環境可用 |
